# Noirven 诺梵 产品需求文档 v0.1

状态：需求梳理阶段，未进入开发。

## 1. 项目定位

Noirven 诺梵是一个独一无二的生成式轻奢艺术首饰 / 私人定制 / 限时拍卖平台。

核心表达：

- 每一次归属，都是一次拯救。
- 不由多数定义，只由唯一确认。
- 独一无二的轻奢质感。
- 极简、好记、差异化，不复制任何现有奢侈品牌的视觉资产。

产品不是传统珠宝商城，而是“灵感生成概念设计 + 后台精选 + 七日拍卖 + 唯一归属”的轻奢展示和交易系统。品牌可以保留 Noirven 的神秘感，但不被“暗夜 / 神圣 / 高奢”单一风格锁死，视觉上更强调简洁、记忆点、现代感和可传播性。

品牌参考方法：

- 学习 Gucci / Hermès / Chanel / Prada 这类国际品牌的极简、强识别、易记忆方法。
- 不复制或近似任何现有品牌 logo、字标、图形、排版比例、经典符号、配色组合或 trade dress。
- 采用 `VoltAgent/awesome-design-md` 的设计文档方法，把视觉规则、组件规则、布局规则和禁忌写入可执行规范。

## 2. 核心业务模式

### 前台

- 展示 Noirven 品牌理念、作品、拍卖、已售作品。
- 用户可注册、登录、出价、支付、查看订单。
- 每件作品独立拍卖，周期固定为 7 天。
- 拍卖到期无人出价，则自动进入下一轮 7 天拍卖。
- 拍卖成交后，作品进入已售归档，显示归属昵称。

### 后台

- 管理员可管理品类、AI 生成设计、作品、拍卖、订单、用户、支付状态和 SEO/GEO 内容。
- 首批品类：戒指、项链、耳环、手环、手表、耳钉。
- 品类后台可新增、编辑、删除、排序、上下架。

## 3. 品牌与作品规则

作品风格要求：

- 轻奢
- 私定
- 精致
- 另类
- 别致
- 独一无二
- 完全自主风格
- 不借鉴或仿照任何现有奢侈品牌风格
- 极简但有记忆点
- 适合社交传播和拍卖展示

主要材质：

- 玫瑰金
- 雕金
- 黄金
- 铂金
- 天然白钻
- 彩色宝石
- 青金石
- 贝母
- 孔雀石

主要制作与着色工艺：

- 珐琅彩
- 钢琴烤漆
- 鎏金
- 鎏彩
- 景泰蓝
- 其他顶级奢侈品级手工制作工艺

作品原则：

- 每件作品对应唯一编号，如 `#N-0001`。
- 每件作品只拍卖一次成交权，不做批量复制。
- 生成式概念图作为概念图 / 渲染展示图，最终实物制作细节需要通过订单和顾问沟通确认。
- 页面必须明确区分：AI 概念图、设计意向、最终交付实物可能存在工艺调整。
- 起拍价范围为 `1888-5999 USD`，系统根据品类、大小、首饰部位、材质、宝石、工艺复杂度和故事系列自动给出建议价，管理员可确认或调整。
- 戒指、手环等涉及尺寸差异的品类，首版设计以活口 / 可调节结构作为默认规格，方便不同买家拍下后自由调节佩戴。
- 每件作品内侧或背面必须刻入 `N+编号`，如 `N-0101`，作为唯一印记、防抄袭识别和证书绑定依据。

## 4. 前台页面结构

### 首页

Hero 文案：

> 每一次归属，都是一次拯救。

核心模块：

- Hero：品牌名、主标语、主视觉、进入拍卖 CTA。
- 当前拍卖：精选 3-6 件。
- 作品归属理念：等待、认领、反转、拯救、唯一性。
- 七日拍卖机制说明。
- 已售作品入口。
- 品牌理念入口。
- 私人顾问 / 联系入口。

### 拍卖页

页面文案：

> 七日期限，等待唯一的主人。

功能：

- 拍卖作品列表。
- 筛选：品类、状态、价格区间、剩余时间。
- 排序：即将结束、最新上架、当前价格。
- 每个作品显示：图片、标题、编号、当前价、剩余时间、出价人数、状态。

### 拍卖详情页

功能：

- 大图展示 / 图集。
- 作品标题、编号、品类、灵感词。
- 作品概念描述。
- 起拍价、当前价、加价幅度、剩余时间。
- 出价入口。
- 历史出价记录，隐藏敏感信息，仅显示昵称脱敏。
- 材质、尺寸、宝石、制作周期等可编辑信息。
- 物流与交付说明。
- 支付说明。
- SEO/GEO 可读的作品说明。

### 已售作品

文案：

> 此件已归于 —— [用户昵称]

功能：

- 已成交作品归档。
- 显示成交价可配置：公开 / 隐藏 / 区间显示。
- 显示归属昵称，需允许买家设置展示昵称或选择匿名。

### 品牌理念

已废弃原始文案，不作为前台主表达：

> Noirven我们生成不可复制的艺术。每一道曲线由AI计算，每一颗宝石的排列都是算法与美学的唯一交汇。没有模具，没有批次，没有第二件，你就是唯一的主人。

最终前台品牌理念文案：

> Noirven 诺梵相信，真正独一无二的作品，不该被流行审判。每一件作品都有自己的编号，进入七日拍卖。有人认领，它从此只归一人；无人出价，它不会被放弃，而是继续等待下一次被看见。每一次成交，都是一次反转：被忽略的被看见，被低估的被证明，曾经等待的人也亲手选择了自己。

## 6. AI 生成工作流

后台流程：

1. 管理员选择品类。
2. 管理员输入灵感词，如“废墟中的凤凰”。
3. 系统组合：
   - Story Lock Prompt
   - Style Lock Prompt
   - 品类 Prompt
   - 灵感词
   - Noirven 禁止仿牌规则
   - 极简轻奢 Style Lock Prompt
4. 使用 OpenAI 图像生成模型生成 4 张概念图。模型名通过环境变量配置，开发前以账号实际可用模型为准。
5. 管理员精选 1 张。
6. 点击“上架拍卖”。
7. 系统辅助生成：
   - 作品标题
   - 作品描述
   - 作品编号，如 `#N-0001`
   - SEO 标题和描述草稿
8. 管理员设定：
   - 起拍价
   - 系统建议起拍价
   - 加价幅度
   - 拍卖开始时间
   - 拍卖周期，默认 7 天
   - 作品材质 / 宝石 / 尺寸 / 工艺说明
   - 所属故事线系列
   - 情绪文案
   - 可调节规格说明
   - 唯一刻印位置

AI 风格锁定要求：

- 不生成任何现有奢侈品牌可识别元素。
- 不出现品牌 logo、商标、经典轮廓或明显借鉴。
- 保持 Noirven 独一无二、极简轻奢、差异化、自主原创的风格。
- 不强制保持暗夜、高奢、神圣孤品作为唯一风格；它们可以作为部分系列的氛围，而不是全站唯一基调。
- 后续产品设计必须复用品牌记忆元素：极简 `N` 印记、作品编号、细横线、轻薄切面、负空间、不完全对称曲线、隐藏刻印位置。
- 详见 `BRAND-LOCK.md`。
- 品牌故事线以 `FINAL-BRAND-STORY.md` 为准，后续产品迭代围绕“等待、被看见、反转、拯救、唯一归属”展开。
- 图像需可用于展示，但页面应标注为 AI 概念图。

## 7. 拍卖规则 v0.1

基础规则：

- 每件作品拍卖周期为 7 天。
- 起拍价由管理员设置。
- 出价必须高于当前价，并满足最低加价幅度。
- 拍卖到期后，最高有效出价者胜出。
- 无人出价则自动开启下一轮 7 天周期。
- 成交后作品状态改为 sold，不再进入拍卖循环。

建议补充规则：

- 拍卖结束前最后几分钟如有新出价，是否自动延时，待确认。
- 是否允许管理员提前下架，待确认。
- 是否允许保留价，待确认。
- 是否允许买家取消出价，建议默认不允许。
- 是否需要防刷出价 / 账户风控，建议需要。

## 8. 支付方案风险与建议

用户提出：

- Stripe。
- BNB 链钱包收款，1 USD = 1 USDT。
- 出价即支付出价金额。
- 竞拍成功即登记发货。
- 竞拍失败原路退回。

### 安全问题

Stripe 私钥已经出现在聊天中，即使是测试密钥，也必须立即去 Stripe 后台作废并重新生成。后续只允许使用环境变量：

- `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`
- `STRIPE_SECRET_KEY`
- `STRIPE_WEBHOOK_SECRET`

私钥不得进入前端、Git、文档、截图或日志。

### Stripe 推荐实现

不建议把每一次出价都立即真实扣款然后失败退款，原因：

- 用户体验差。
- 退款频繁。
- 手续费、风控、争议和对账复杂。
- 7 天周期可能触及银行卡授权窗口限制。

更稳妥的方案有三种：

1. 保证金模式：出价时支付或预授权固定比例保证金，胜出后支付尾款。
2. 最高出价预授权模式：只对当前最高出价者保留授权，被超越后释放授权。
3. 立即扣款 + 失败退款模式：最符合原始想法，但风险最高，不建议作为正式版默认方案。

当前建议：

- MVP 只在两种方案中选择：
  - 方案 A：保证金 + 成交尾款。
  - 方案 B：最高出价预授权。
- 不建议 MVP 采用“出价即全额扣款，失败后退款”。
- 默认推荐方案 A，因为 7 天拍卖周期下更容易处理支付授权过期、退款、争议和跨支付方式结算。

### BNB / USDT 推荐实现

BNB 链 USDT 支付需要明确：

- 使用 BEP-20 USDT。
- 固定收款地址。
- 用户提交交易哈希。
- 后端校验收款地址、金额、币种、确认数和时间。
- 不建议自动退款，除非后续接入安全托管的钱包签名服务。

MVP 建议：

- USDT 可以作为成交尾款或结算方式加入 MVP。
- 先做 USDT 手动/半自动确认。
- 后台审核交易哈希。
- 退款由管理员人工处理并记录。
- Stripe 仍作为保证金或预授权的主路径。

## 9. 订单与交付

成交后流程：

1. 拍卖结束。
2. 系统锁定胜出者。
3. 支付确认。
4. 买家补充或确认收货信息。
5. 管理员创建订单。
6. 作品进入制作 / 备货 / 发货流程。
7. 买家可查看状态。

订单状态建议：

- `awaiting_payment`
- `paid`
- `awaiting_shipping_info`
- `in_production`
- `quality_check`
- `shipped`
- `delivered`
- `cancelled`
- `refunded`

## 10. 技术栈推荐

推荐主栈：

- Frontend: Next.js App Router + TypeScript。
- Styling: Tailwind CSS + 自定义 Noirven 设计系统。
- UI 基础组件: Radix UI 或 shadcn/ui，只用无障碍底层组件，不套普通 SaaS 风格。
- Animation: Framer Motion，少量克制动效。
- Backend: Supabase Auth + Postgres + Storage + Edge Functions。
- Auction Engine: Supabase Postgres transaction + row-level lock。
- Scheduled Jobs: Supabase Cron / pg_cron 调用 Edge Functions。
- Payments: Stripe PaymentIntents + Webhooks；MVP 默认“保证金 + 成交尾款”或“最高出价预授权”；USDT 作为尾款 / 结算方式走链上校验或人工审核。
- AI: OpenAI GPT Image 系列。用户期望使用 `gpt-image-2`，但需开发前按官方文档和账号权限确认实际可用模型名。
- Deployment: Vercel 前端 + Supabase 后端。
- Monitoring: Supabase logs + Vercel logs，后续可加 Sentry。

技术原则：

- 所有支付、AI、管理员敏感操作都走后端函数。
- 前端永远不能接触 Stripe 私钥、OpenAI Key、Supabase Service Role Key。
- 所有核心写操作必须幂等。
- 所有竞拍操作必须有数据库事务、锁和审计日志。
- SEO 页面优先服务端渲染或静态生成。

## 11. 数据模型草案

核心表：

- `profiles`
- `categories`
- `story_series`
- `design_generations`
- `generated_images`
- `products`
- `auctions`
- `bids`
- `payments`
- `orders`
- `shipping_addresses`
- `refunds`
- `admin_audit_logs`
- `site_content`
- `seo_metadata`

关键约束：

- 一个 product 可对应多个 auction cycle，但成交后停止循环。
- 一个 product 必须绑定一个 story_series，首批系列为 Still Here、Unclaimed Star、Seventh Light、Justice of One。
- 一个 story_series 包含主题、情绪文案、视觉记忆元素、IP 延展说明和排序字段。
- 一个 auction 有多个 bids。
- 一个 bid 必须绑定 payment 状态。
- 一个 sold product 只能有一个最终 order。
- 所有管理员操作写入 audit log。

## 12. SEO / GEO 同步要求

每个页面上线时必须同步：

- title
- description
- canonical

- H1 / H2
- 图片 alt
- JSON-LD
- sitemap

拍卖作品页需要结构化描述：

- 为什么是唯一作品。
- 拍卖何时结束。

## 13. 需要确认的问题

1. 拍卖结束前最后 5 分钟有人出价，是否自动延长 5 分钟？
2. 拍卖是否设置保留价？
3. 成交价格是否公开显示在已售作品页？
4. 买家是否可以匿名展示为“Private Collector”？
5. MVP 支付模式最终选哪种：方案 A“保证金 + 成交尾款”，还是方案 B“最高出价预授权”？
6. USDT 是否作为首版成交尾款 / 结算方式上线？
7. 首版是否需要真实 3D 模型，还是先用 AI 2D 概念图 / 渲染图？
8. 管理员是否需要多账号和权限分级？
9. 采用多语言策略，首版支持中文和英文。
10. 面向中国及海外市场，需要在支付、物流、隐私政策、SEO/GEO、货币展示和客服内容上预留区域差异。
